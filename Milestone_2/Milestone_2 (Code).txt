\section{Introduction}

\subsection{Purpose}
\label{subsec:purpose}

This Software Requirements Specification (SRS) document provides a detailed description of all functions and specifications of the Smart Hostel Management System (SHMS). This (SRS) document is intended for:

\begin{itemize}
    \item \textbf{Hostel Administration:} Ms. Nida Sultan Nahra (Hostel Warden) and hostel management staff who will use the system to manage daily hostel operations, and those operations are like monitoring student attendance, getting online complaints by students and allocation of the rooms.
    
    \item \textbf{Development Team:} The software development team (Muhammad Ahmad, Asad Ullah Khan, and Maryam Rashid) who will design, implement, test, and maintain the system based on the requirements specified in this document and discussed with their RP.
    
    \item \textbf{Students (End Users):} Hostel residents who will interact with the system to mark attendance, sumbit online reason before leaving Hostel/University, register complaints, group wise room selection and manage their hostel-related activities.

    On the other Hand Day Scholars (student) will also included in the attendance, in order to make proper check in that who enter hostel or when exit Hostel/University.
    
    \item \textbf{System Administrators:} Technical staff will be responsible for deploying, configuring, and maintaining the system infrastructure in the Hostel as well as in the university.
    
    \item \textbf{Quality Assurance Team:} Testing will be done by the team to verify that the implemented system meets all specified requirements and quality standards.
    
    \item \textbf{Stakeholders:} Hostel Management, University management, security personnel, maintenance staff who interact with or are affected by the system.
\end{itemize}

The purpose of SHMS is to automate and digitize hostel management operations at Namal University to replace the manual and inconsistent processes with a unified, efficient, and transparent digital platform.

\subsection{Scope}
\label{subsec:scope}

The Smart Hostel Management System (SHMS) is a comprehensive web-based application designed to manage all aspects of hostel operations at Namal University. The system is designed for female hostel facilities and operates within the girls hostel environment and as well as in the university. It enforces strict security and access controls to ensure that only authorized users can access system functions and student information. All data related to hostel residents is handled securely to maintain privacy and safety at all times.

\subsubsection{System Name and Components}

\textbf{System Name:} Smart Hostel Management System (SHMS)

\textbf{Major Components:}
\begin{itemize}
    \item Attendance Management Module
    \item Biometric Entry and Exit Tracking System
    \item Room Allocation and Management Module
    \item Day Scholar Tracking Module
    \item Centralized Student Database
    \item Reporting and Analytics Dashboard
    \item Complaint Management System
\end{itemize}

\subsubsection{What the Software Will Do}

The SHMS will accomplish the following objectives:

\begin{enumerate}
    \item \textbf{Automate Attendance Tracking:}
    \begin{itemize}
        \item Integrate with the existing biometric systems to automatically record student attendance for hostelers using biometric devices placed inside the hostel
        \item Provide real-time attendance dashboards for wardens
        \item Automatically identify and flag absent students
        \item Generate daily, weekly, and monthly attendance reports
        \item Send automated notifications (through mail or SMS) to wardens at 10:05 PM about unauthorized absences and absentee list
        \item Link attendance with the last online exit application submitted by the student
    \end{itemize}
    
    \item \textbf{Biometric Entry and Exit Tracking:}
    \begin{itemize}
        \item Enable students to submit online reason/application before exiting hostel or university (no approval needed except for restricted times after 5 PM)
        \item Provide hostel warden with a centralized interface to monitor and view entry/exit logs in real time.
        \item Automatically records each student during entry/exit without the need for manual gate passes using biometric devices at main gate
        \item Allow security personnel to make sure that students are using the biometric system during entry/exit points.
        \item Track student return status and send automated notification for late returns or violations (immediate for restricted time violations)
        \item For short leaves (shopping, medical, etc.), exit allowed only before 5 PM and entry compulsory before 5 PM, with immediate notifications for violations
        \item For day scholars, track entry/exit from hostel and university (application required for non-routine university exit and no approval needed)
        \item Send daily notifications at 10:05 PM to warden about attendance, day scholars in hostel, and any security events
    \end{itemize}
    
    \item \textbf{Streamline Room Management:}
    \begin{itemize}
        \item Maintain real-time inventory of approximately 50 rooms with capacities of 2, 4, or 8 beds
        \item Facilitate group-wise room allocation where exactly 2, 4, or 8 students agree and one submits the form with all details; no duplicate registrations allowed
        \item Track room occupancy history and maintenance schedules
        \item Manage room change requests with reason and vacancy viewing
        \item Handle room clearance processes when students vacate
        \item Generate room occupancy reports and statistics
    \end{itemize}
    
    \item \textbf{Handle Complaints:}
    \begin{itemize}
        \item Allow students to submit online complaints with details and attachments
        \item Send immediate notifications to warden
        \item Track status and resolution
        \item Generate complaint reports
    \end{itemize}
    
    \item \textbf{Monitor Day Scholar Access:}
    \begin{itemize}
        \item Register and manage day scholar profiles
        \item Track day scholar entry and exit from hostel premises using biometrics
        \item Send real-time notifications when day scholars enter hostel
        \item Maintain logs for day scholars
        \item Include current day scholar count in daily 10:05 PM notification
    \end{itemize}
    
    \item \textbf{Centralize Student Information:}
    \begin{itemize}
        \item Maintain unified student profiles with hostel-related information
        \item Synchronize with university student information systems
        \item Provide role-based access to student data
        \item Generate comprehensive student hostel history reports
    \end{itemize}
\end{enumerate}

\subsubsection{Benefits and Objectives}

The implementation of SHMS will provide the following benefits:

\begin{itemize}
    \item \textbf{Efficiency:} Reduce manual paperwork and administrative load by 70-80\% and allowing wardens to focus on student welfare rather than administrative tasks.
    
    \item \textbf{Transparency:} Provide real-time visibility into all hostel operations for authorized stakeholders and ensuring that everyone has equal access to system.
    
    \item \textbf{Security:} Enhance hostel security through digital verification on gate entry/exit through biometric, automated absence tracking, and day scholar monitoring, with focus on girls' security.
    
    \item \textbf{Accuracy:} Eliminate human errors in attendance marking or finding who was absent and help a lot in record keeping.
    
    \item \textbf{Accessibility:} Enable 24/7 access to hostel services and information through web interfaces.
    
    \item \textbf{Data-Driven Decision Making:} Offer detailed reports and analytics on student activities and hostel operations to guide effective management and improving overall hostel efficiency.
\end{itemize}

\subsubsection{What the Software Will NOT Do}

To set clear boundaries, the following are explicitly outside the scope of SHMS:

\begin{itemize}
    \item The system will not handle academic performance tracking or course registration (these remain with the university's academic management system)
    
    \item The system will not process financial aid or scholarship applications (handled by university financial services)
    
    \item The system will not manage library operations, sports facilities, or other non-hostel university services
    
    \item The system will not replace or integrate with existing university ERP for financial accounting
    
    \item The system will not provide medical services management or health records (it will be handled by university health center)
    
    \item The system will not manage faculty or staff housing (scope limited to girls student)
    
    \item The system will not handle disciplinary actions or conduct violations (these remain administrative decisions documented separately)
\end{itemize}

\section{General Description}
\subsection{Product Perspective}
\label{sec:product-perspective}
The Smart Hostel Management System (SHMS) is a new custom-developed web-based application which wll be used for Namal University's hostel operations. It addresses limitations in the current minor digital system, such as marking attendance but doesn't know who is missing, time-consuming verifications, and lack of real-time reporting. SHMS represents an evolution from previous tools (e.g., biometric devices for girls' hostel to track in/out) to an integrated platform.
\subsubsection{Background and Current System Limitations}
Currently, hostel management relies on manual processes: attendance is recorded via biometrics for girls but manually checked who marked attendance, gate passes are paper-based, room allocations are tracked in spreadsheets, and day scholar tracking is ad-hoc. These actions lead to errors, delays, and security risks. SHMS aims to centralize these into a digital system, reducing workload and improving accuracy.
\subsubsection{System Interfaces}
SHMS will integrate with:
\begin{itemize}
\item \textbf{Biometric Attendance Systems:} Real-time data import from fingerprint devices.
\item \textbf{University Student Database:} API access for student profiles, enrollment, and updates.
\item \textbf{Email/SMS Gateways:} For alert notifications.
\end{itemize}
\subsubsection{User Interfaces}
Responsive and user-friendly web UI with dashboards, forms, and reports. Supports accessibility features like keyboard navigation.
\subsubsection{Hardware Interfaces}
\begin{itemize}
\item Biometric devices on main gates.
\item Server hardware or cloud (e.g., AWS/EC2 equivalents).
\item Client devices: Desktops/Laptops, mobiles with browser support.
\end{itemize}
\subsubsection{Software Interfaces}
\begin{itemize}
\item OS: Windows OR Mobile browser.
\item DBMS: MongoDB for data storage.
\item Framework: Node.js/Express for backend, React.js for frontend.
\item Browsers: Chrome, Firefox, etc.
\end{itemize}
\subsubsection{Communications Interfaces}
\begin{itemize}
\item HTTPS for secure data.
\item SMTP for emails.
\item RESTful APIs for internal/external communication.
\item Optional SMS via third-party gateway.
\end{itemize}
\subsubsection{System Context}
SHMS operates within hostel boundaries, aligned with university calendars, rules, and privacy policies. It interacts with external entities like students, staff, and IT systems.
\subsection{Product Functions}
\label{sec:product-functions}
SHMS has main modules to handle daily hostel operations.
\subsubsection{Attendance Management}
This module takes care of daily attendance to keep track of student presence and safety. It supports biometric scanning, manual entry when needed, leave requests, automatic absentee lists, and helpful reports and alerts for staff.
\subsubsection{Biometric Entry and Exit Tracking}
This module handles student entry and exit from the hostel and university using digital verification. Students can apply online, upload documents if needed, get approvals from staff for restricted times, and security personnel verify at the gate to record actual exit and return times. The system also sends alerts for overdue returns and provides reports.
\subsubsection{Room Management}
This module helps assign and manage hostel rooms. It keeps a record of all rooms, suggests or assigns rooms to new students, handles transfer requests, tracks maintenance issues, and manages room clearance when students leave.
\subsubsection{Complaint Management}
This module looks after student complaints. Students can file complaints (with photos), and the system tracks each complaint, sending notifications to warden.
\subsubsection{Day Scholar Tracking}
This module controls who enters the hostel. Security staff can log day scholars, get approvals if needed, issue time-limited passes, record entry and exit times, and send alerts if someone stays too long.

\subsection{General Constraints}

This section explains the limitations and conditions that may affect how the Smart Hostel Management System (SHMS) is designed, developed, and used. These constraints must be considered to ensure that the system functions effectively within the university environment.

\subsubsection{Regulatory and Policy Constraints}

\begin{itemize}
    \item The system shall comply with all university rules and policies related to attendance, hostel operations, and student privacy.
    \item The system shall maintain female hostel data in such a way to ensure privacy and security.
    \item The system shall protect all stored student information according to data protection and privacy guidelines.
\end{itemize}

\subsubsection{Hardware Limitations}

\begin{itemize}
    \item The system shall operate with the existing biometric devices installed in the hostels, even if these devices have limited features or outdated APIs.
    \item The system's performance may depend on the speed and stability of the universityâ€™s network infrastructure.
    \item The system shall run effectively on a variety of staff devices, including desktops, laptops, tablets, and mobile phones.
    \item The system's availability may be affected by power interruptions, and it must rely on the university's UPS or backup systems.
\end{itemize}

\subsubsection{Interface Requirements}

\begin{itemize}
    \item The system shall integrate with the existing student information database through approved APIs or supported data formats.
    \item The system shall use standard web communication protocols to ensure compatibility across different devices and networks.
    \item The system shall follow basic accessibility guidelines to support users with different abilities.
\end{itemize}

\subsubsection{Development Constraints}

\begin{itemize}
    \item The system shall be fully developed, tested, and deployed within the limited academic semester timeline.
    \item The system shall be developed by a small student team, requiring the design to match their technical skill level.
    \item The system shall be built using free or open-source technologies to avoid licensing costs.
    \item The system shall use the predefined tech stack (React.js, Node.js, MongoDB) for consistency across all modules.
\end{itemize}

\subsubsection{Operational Constraints}

\begin{itemize}
    \item The system shall remain available during major hostel operating hours, with occasional maintenance windows when needed.
    \item The system shall support a large number of users at the same time, especially during busy evening periods.
    \item The system shall provide limited offline support to ensure essential features remain accessible during slow or unstable internet conditions.
    \item The system's performance may vary during peak usage times, such as semester start or hostel room allocation days.
\end{itemize}

\subsubsection{Security and Safety Constraints}

\begin{itemize}
    \item The system shall use role-based access control to ensure that only authorized users can perform sensitive operations.
    \item The system shall maintain secure audit logs of important user actions, such as approvals and data updates.
    \item The system shall implement measures to protect against unauthorized access, data breaches, and cyber threats.
    \item The system shall support emergency procedures by allowing wardens quick access to attendance and gate pass information when needed.
\end{itemize}

\section{Specific Requirements}
This section details requirements, organized by external interfaces and functional modules. Performance requirements are included per module.
\subsection{External Interfaces}
\label{sec:external-interfaces}
\subsubsection{User Interfaces}
The system shall have a clean and responsive web interface built with React.js. It will show different dashboards based on the user's role, easy forms (for example, an entry/exit request form with fields for dates and reason), and reports that can be downloaded as PDF or CSV. The interface will be simple to use, with helpful hints, clear error messages, and full support for mobile devices.
\subsubsection{Hardware Interfaces}
The system shall connect to biometric fingerprint devices through network APIs. The server needs at least 8GB RAM and a 4-core CPU (can be university server or cloud). Users can access the system from any computer, tablet, or phone with a modern web browser.
\subsubsection{Software Interfaces}
The system shall use MongoDB to store all data with proper structure for students, rooms, attendance, etc. The backend will run on Node.js with Express to handle APIs (for example, POST requests for saving attendance). The frontend will use React.js for smooth and fast user interaction. It will work on Linux servers and be compatible with all major web browsers.
\subsubsection{Communications Interfaces}
The system shall always use secure HTTPS for all connections. It will communicate using REST APIs with JSON data format. Emails will be sent via SMTP. The system will work over the university's normal internet or any Wi-Fi connection.
\subsection{Attendance Management Module}
This module ensures accurate tracking of student presence, integrating biometrics for automation. It includes workflows for leaves and reports to support warden decision-making.
\subsubsection{Functional Requirements}
\paragraph{ATT-REQ-001: Biometric Data Import}
The system shall automatically import attendance records from biometric devices every hour between 6:00 PM and 11:59 PM daily, validating data for duplicates and logging errors.\\
Description: This feature pulls data via API, maps fingerprints to student IDs, and updates the database.
\paragraph{ATT-REQ-002: Manual Attendance Entry}
The system shall allow wardens and assistant wardens to manually mark attendance for students when biometric systems are unavailable.\\
Description: UI form with student search, date selection, and confirmation.
\paragraph{ATT-REQ-003: Attendance Deadline}
The system shall consider all students who have not marked attendance by 10:00 PM as absent for that day, triggering automated processes.\\
Description: Cron job runs at deadline to flag absentees.
\paragraph{ATT-REQ-004: Absentee Identification}
The system shall automatically generate a list of absent students at 10:01 PM each day, excluding students with approved leave applications, and sort by room/floor.\\
Description: Query database for unmarked records.
\paragraph{ATT-REQ-005: Absentee Notification}
The system shall send automated notifications to wardens and assistant wardens within 5 minutes of generating the absentee list, including list attachments.\\
Description: Email with links to dashboard.
\paragraph{ATT-REQ-006: Leave Request Submission}
The system shall allow students to submit leave requests at least 24 hours in advance, specifying departure date, return date, reason, and optional attachments (max 5MB).\\
Description: Form validation for dates, reason length.
\paragraph{ATT-REQ-007: Leave Request Approval Workflow}
The system shall route leave requests through a two-level approval process: first to assistant warden, then to warden, with email notifications at each step.\\
Description: State machine for status (pending, approved1, approved2).
\paragraph{ATT-REQ-008: Emergency Leave Request}
The system shall provide a mechanism for emergency leave requests that require only warden approval and can be submitted with less than 24 hours notice, flagged as high priority.\\
Description: Separate form with urgency field.
\paragraph{ATT-REQ-009: Leave Status Notification}
The system shall notify students via email within 5 minutes when their leave request status changes (approved, rejected, or pending), including reasons for rejection.\\
Description: Webhooks for real-time updates.
\paragraph{ATT-REQ-010: Attendance History Viewing}
The system shall allow students to view their own attendance history for the current semester and previous semesters, with filters and export options.\\
Description: Paginated table with dates, status.
\paragraph{ATT-REQ-011: Attendance Report Generation}
The system shall allow wardens to generate attendance reports filtered by date range, hostel wing, floor, room number, or individual student, in PDF/CSV format.\\
Description: Custom queries with aggregation.
\paragraph{ATT-REQ-012: Attendance Statistics Dashboard}
The system shall display real-time attendance statistics including total present, total absent, leave count, and attendance percentage on the warden dashboard, with charts.\\
Description: Live updates via WebSockets.
\paragraph{ATT-REQ-013: Repeated Absence Alerts}
The system shall automatically flag students who have been absent for 3 or more consecutive days without approved leave and send alerts to the warden, with student details.\\
Description: Daily check job.
\paragraph{ATT-REQ-014: Attendance Data Retention}
The system shall retain attendance records for a minimum of 5 years for historical reference and compliance purposes, with archiving for old data.\\
Description: Database partitioning.
\paragraph{ATT-REQ-015: Bulk Attendance Import}
The system shall support bulk import of attendance data from CSV or Excel files for migration or backup restoration purposes, with validation.\\
Description: Admin-only feature.
\paragraph{ATT-REQ-016: Attendance Correction}
The system shall allow wardens to correct attendance records within 48 hours, requiring justification and audit logging.\\
Description: Edit form with history.
\paragraph{ATT-REQ-017: Integration with University Calendar}
The system shall integrate with university holidays to adjust attendance expectations automatically.\\
Description: API pull for dates.
\paragraph{ATT-REQ-018: Student Attendance Reminders}
The system shall send reminders to students at 9:00 PM if attendance is not marked.\\
Description: Scheduled notifications.
\paragraph{ATT-REQ-019: Export Attendance Data}
The system shall allow export of attendance data for external analysis.\\
Description: Multiple formats.
\paragraph{ATT-REQ-020: Attendance Audit Logs}
The system shall maintain logs of all attendance changes for security.\\
Description: Immutable records.
\subsubsection{Performance Requirements}
\paragraph{ATT-PERF-001}
The system shall process biometric imports for 1000 students within 1 minute.
\paragraph{ATT-PERF-002}
Absentee generation shall complete within 10 seconds for 500 students.
\paragraph{ATT-PERF-003}
Reports shall generate within 10 seconds for 1 month data.
\paragraph{ATT-PERF-004}
Dashboard updates shall occur in real-time (<2 seconds lag).

\section{Other Non-Functional Requirements}
\subsection{Security Requirements}

The following security requirements ensure that the Smart Hostel Management System remains safe and protects all user data:

\begin{itemize}

    \item \textbf{SEC-REQ-001:} The system shall restrict access based on user roles so that each user can only view or use the features allowed for their position.

    \item \textbf{SEC-REQ-002:} The system shall protect all user accounts with secure login and strong password rules to prevent unauthorized access.

    \item \textbf{SEC-REQ-003:} The system shall encrypt sensitive information, including student records, attendance data, and entry or exit tracking, both during storage and transmission.

    \item \textbf{SEC-REQ-004:} The system shall maintain logs of important actions, such as approvals or data updates, to help track suspicious or incorrect behavior.

    \item \textbf{SEC-REQ-005:} The system shall include protections against common security threats such as brute-force attacks, unauthorized page access, and harmful code injections.

\end{itemize}

\subsection{Reliability Requirements}

The following reliability requirements ensure that the Smart Hostel Management System performs consistently and remains dependable during everyday use:

\begin{itemize}

    \item \textbf{REL-REQ-001:} The system shall remain available and fully operational during standard hostel hours (6:00 AM to 11:00 PM) with an uptime target of 99\%.

    \item \textbf{REL-REQ-002:} The system shall recover from unexpected server failures within a few minutes without losing any previously saved or committed data.

    \item \textbf{REL-REQ-003:} The system shall handle high user load smoothly, ensuring stable performance even when many students or staff use the system at the same time.

    \item \textbf{REL-REQ-004:} The system shall safely store user actions such as attendance updates so that no data is lost during temporary network interruption.

    \item \textbf{REL-REQ-005:} The system shall automatically create regular data backups to ensure that information can be restored in case of technical failure.

    \item \textbf{REL-REQ-006:} The system shall operate reliably across multiple device types (mobile, desktop, tablet) without unexpected crashes or errors.

\end{itemize}

\subsection{Usability Requirements}

The following usability requirements ensure that the Smart Hostel Management System is simple, comfortable, and easy for all types of users:

\begin{itemize}
    
    \item \textbf{USA-REQ-001:} The system shall support both English and Urdu languages so users can choose the language they are most comfortable with.
    
    \item \textbf{USA-REQ-002:} The system shall allow new users to perform basic tasks, such as submitting attendence, without requiring prior training.
    
    \item \textbf{USA-REQ-003:} The system shall provide tooltips and help icons on detailed or complex features to guide users when needed.
    
    \item \textbf{USA-REQ-004:} The system shall display error messages in simple, user-friendly language along with suggestions on how to fix the issue.
    
    \item \textbf{USA-REQ-005:} The system shall maintain a consistent color scheme, layout, and visual design across all pages to avoid user confusion.
    
    \item \textbf{USA-REQ-006:} The system shall clearly highlight important actions such as â€œSubmit,â€ â€œSave,â€ and â€œApproveâ€ to help users complete tasks easily.
    
\end{itemize}

\subsection{Supportability Requirements}

The following supportability requirements ensure that the Smart Hostel Management System can be easily maintained, updated, and supported after deployment:

\begin{itemize}
    \item \textbf{SUP-REQ-001:} The system shall be easy to update so that new features or fixes can be added without affecting other modules.
    
    \item \textbf{SUP-REQ-002:} The system shall include clear documentation that explains how each part of the system works and how it can be maintained.
    
    \item \textbf{SUP-REQ-003:} The system shall provide helpful error messages and maintain proper logs to assist technical staff in troubleshooting issues quickly.
    
    \item \textbf{SUP-REQ-004:} The system shall be designed in a modular structure so different modules (attendance, gate pass, rooms, etc.) can be maintained separately.
    
    \item \textbf{SUP-REQ-005:} The system shall support regular data backups to ensure easy recovery of important information if any failure occurs.
    
    \item \textbf{SUP-REQ-006:} The system shall be designed to support future improvements and upgrades with minimal changes.
\end{itemize}

\subsection{Portability Requirements}

The portability requirements ensure that the Smart Hostel Management System can run easily across different devices, browsers, and hosting environments. The system should remain usable even if the university changes its technical setup in the future.

\begin{itemize}
    \item \textbf{PORT-REQ-001:} The system shall run smoothly on all major web browsers including Chrome, Firefox, Edge, and Safari without requiring special configurations.
    
    \item \textbf{PORT-REQ-002:} The system shall be usable on multiple device types such as desktops, laptops, tablets, and smartphones with consistent functionality.
    
    \item \textbf{PORT-REQ-003:} The system shall be operating-system independent and function properly on Windows, macOS, and Linux platforms.
    
    \item \textbf{PORT-REQ-004:} The system shall be easily transferable to another server or hosting environment if required by the university.
    
    \item \textbf{PORT-REQ-005:} The system shall support cloud-based deployment on platforms like AWS, Azure, or any university-approved cloud service.
    
    \item \textbf{PORT-REQ-006:} The system shall use standard web technologies so that future updates or migrations can be performed without major code modifications.
\end{itemize}

\subsection{Maintainability Requirements}

The maintainability of the Smart Hostel Management System is important to ensure that the system can be updated, improved, and fixed easily over time. The following non-functional requirements define how maintainability will be supported:

\begin{itemize}
    \item \textbf{MAIN-REQ-001:} The system shall be developed using a modular structure so that each major feature (attendance, Biometric Tracking system, room management, complaints) can be maintained independently.
    
    \item \textbf{MAIN-REQ-002:} The system shall be designed in a way that allows new features or policy changes to be added with minimal changes to the existing code.
    
    \item \textbf{MAIN-REQ-003:} The system shall include clear and simple documentation that explains modules, workflows, and important functions to help future developers maintain the system easily.
    
    \item \textbf{MAIN-REQ-004:} The code-base shall follow consistent naming conventions and folder structures so developers can quickly locate and update required sections.
    
    \item \textbf{MAIN-REQ-005:} The system shall maintain readable logs and organized error messages to support quick debugging and problem resolution.
    
    \item \textbf{MAIN-REQ-006:} The system shall store adjustable settings (such as attendance deadlines, visiting hours) in configuration files instead of hard-coded values.
    
    \item \textbf{MAIN-REQ-007:} The database design shall be flexible enough to allow the addition of new fields, tables, or relationships as hostel needs grow over time.
    
    \item \textbf{MAIN-REQ-008:} The system shall support maintenance operations with minimal downtime to ensure that updates or fixes do not disturb daily hostel activities.
\end{itemize}
